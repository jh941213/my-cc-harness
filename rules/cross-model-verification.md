# 교차 모델 검증 규칙

## 핵심 원칙
**같은 모델이 만든 것을 같은 모델이 검증하지 않는다.**

## 계획 단계 교차 검증

계획(Plan) 수립 시 3단계 이상이면 반드시 Codex에게 계획 검토를 요청한다:

```
1. [Claude] 계획 초안 작성
2. [Codex] 계획 검토 — 빠진 엣지케이스, 과도한 스코프, 순서 오류 지적
3. [Claude] 피드백 반영하여 계획 확정
4. 사용자 승인 대기
```

### Codex 계획 검토 프롬프트 템플릿
```
다음 구현 계획을 검토해줘:
[계획 내용]

확인 사항:
- 빠진 엣지케이스나 의존성은?
- 단계 순서가 최적인가?
- 스코프가 과도하거나 부족한 부분은?
- 검증 커맨드가 충분한가?
- 보안/성능 관점에서 놓친 것은?
```

## 코드 검증 교차 루프

```
1. [Claude] 코드 1차 작성
2. [Codex] 1차 검증 + 수정 제안
3. [Claude] 수정 반영 + 2차 검증
4. [Codex] 최종 검증 → PR 단계
```

## 자기 평가 금지

- 같은 세션에서 코드를 짜고 같은 모델에게 "평가해봐" → **금지**
- eval은 반드시 별도 세션 또는 별도 모델(Codex)에서 실행
- evaluator 에이전트는 구현 에이전트와 완전히 분리

## 적용 조건

| 상황 | 교차 검증 필요 | 생략 가능 |
|------|---------------|----------|
| 3단계 이상 계획 | O — Codex 검토 필수 | X |
| 아키텍처 변경 | O — Codex + 사람 검토 | X |
| 1-2줄 수정 | X | O |
| 버그 수정 (원인 명확) | X | O |
| 테스트만 추가 | X | O |
