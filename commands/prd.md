# PRD 생성 — 아이디어 → 기획 산출물

사용자의 아이디어를 분석하여 PRD + SPEC을 한 번에 생성합니다.
`/tth`로 바로 이어갈 수 있는 완전한 기획 산출물을 만듭니다.

입력: $ARGUMENTS

## 산출물 구조

```
prd/
├── CPS.md          ← 세계관 정렬 (Context-Problem-Solution)
├── PRD.md          ← 코어 전략 (한줄정의 + 인사이트 + 전략 + MVP)
├── MARKET.md       ← 경쟁사 + 오픈소스 + 시장 지형
├── USERS.md        ← 페르소나 + 니즈 + "말 vs 행동"
├── FEATURES.md     ← 기능별 유저스토리 + 인수조건 + MoSCoW
├── RISKS.md        ← 위험 + 완화 + 성공지표 + 피봇기준
├── SPEC.md         ← 기술 명세 (스택 + API + DB + 아키텍처)
└── APPENDIX.md     ← 수렴보드 + 리서치출처 + 인터뷰요약 + 용어
```

---

## Phase 0: 입력 수신 + 복잡도 게이트

**먼저 확인**: `prd/` 디렉토리가 존재하면 → "이전 세션을 이어서 진행할까요?" 질문.
존재하지 않으면 새로 시작.

사용자 입력을 받으면 **복잡도를 판정**하고 수렴 보드를 초기화:

```
Low  (단순 기능/도구/CLI): Phase 1 스킵, Phase 2를 2라운드로 축소, SPEC 간소화
Mid  (새 모듈/서비스/라이브러리): Phase 1 축소(서브에이전트 1개), Phase 2를 3라운드
High (새 제품/SaaS/플랫폼): 전체 플로우 실행
```

복잡도를 판정한 뒤 사용자에게 고지. 조정 요청 시 수용.

**수렴 보드 초기화**:
```
| 차원 | 상태 | 잔여 |
|------|------|------|
| CPS 정렬 | 🔴 | - |
| 용어 정렬 | 🔴 | - |
| 구조 (W6H) | 🔴 | 7개 미확인 |
| 깊이 (가정 검증) | 🔴 | - |
| 일관성 (모순) | 🔴 | - |
| 견고성 (반론) | 🔴 | - |
| 시장 컨텍스트 | 🔴 | - |
```

## Phase 0.5: CPS (Context-Problem-Solution) 문서화

인터뷰 시작 전에, 사용자의 입력으로부터 CPS 프레임을 빠르게 정리한다.
CPS는 PRD보다 상위 레벨 — 프로젝트의 "세계관 정렬" 역할.

**사용자에게 AskUserQuestion으로 확인:**

1. **Context (맥락)**: "현재 어떤 상황/환경에서 이 프로젝트가 필요한가요?"
2. **Problem (문제)**: "구체적으로 어떤 문제를 겪고 있나요? 정량적 지표가 있다면?"
3. **Solution (해결책)**: "이 프로젝트가 성공하면 어떤 상태가 되어야 하나요?"

CPS 확정 후 `prd/CPS.md`에 즉시 저장:

```markdown
# CPS (Context-Problem-Solution)

## Context
[누가, 어디서, 어떤 환경에서]

## Problem
[구체적 문제 + 정량적 임팩트]

## Solution
[해결 후 기대 상태 + 성공 지표]
```

수렴 보드 업데이트: `CPS 정렬 → 🟢 확정됨`

## Phase 1 + Phase 2 라운드 1: 병렬 시작

**동시에 실행** (라운드 1은 리서치 결과 불필요):

### [병렬 A] 시장 리서치 서브에이전트 (background: true)

복잡도별 적응:
- **High**: 서브에이전트 3개 병렬
- **Mid**: 서브에이전트 1개
- **Low**: 스킵

**서브에이전트 A: 경쟁사 & 유사 제품** (subagent_type: general-purpose)
- Tavily + Gemini CLI로 유사 제품/서비스 5-10개 검색 (교차 검증)
- 핵심: **모든 경쟁사가 공통으로 놓치는 것** 식별

**서브에이전트 B: 오픈소스 & 기술 생태계** (subagent_type: general-purpose)
- Exa로 GitHub 유사 프로젝트 + Gemini CLI로 기술 트렌드 보완
- 핵심: **오픈소스의 한계점 = 우리의 기회** 도출

**서브에이전트 C: 사용자 페인포인트 & 시장 동향** (subagent_type: general-purpose)
- Tavily + Gemini CLI로 Reddit, HN, 커뮤니티 불만/니즈 검색
- 핵심: **반복되는 불만 패턴** 및 **미해결 니즈** 추출

### [병렬 B] 라운드 1: 용어 정렬 + 구조 스캔 (메인 세션)

리서치 결과 없이 즉시 시작:
- **[용어]** 모호한 용어 탐지 → 정의 강제
- **[구조:W6H]** 7개 차원 중 비어 있는 것을 AskUserQuestion으로 질문
- **수렴 조건**: 핵심 용어 lock + W6H 7/7

## Phase 2: 적응적 인터뷰 (라운드 2부터, 메인 세션)

리서치 결과 합류 후 계속. 하드캡: Low 2라운드, Mid 3라운드, High 5라운드.

| 라운드 | 프레임워크 | 목적 | 수렴 조건 |
|--------|-----------|------|-----------|
| 2 | 방법적 회의 + White/Red Hat | 가정 검증 + 깊이 | 핵심 차원 🟢, 🟡≤2 |
| 3 | 모순 탐지 + Black/Green Hat | 모순 + 맹점 | 치명적 모순 0개 |
| 4 | 반론 단련 + Yellow/Blue Hat | 반론-대응 쌍 | 반론-대응 쌍 기록 |
| 5 | 메타 정리 (High만) | 우선순위의 진실 | 수렴 보드 전체 🟢 |

Low → 라운드 2 후 종료, Mid → 라운드 3 후 종료.

### 인터뷰 운영 규칙

- **질문 태그**: `[구조:Who]`, `[회의:가정]`, `[모순]`, `[반론]`, `[정합성]`, `[맹점]`
- **레이어 병합**: 한 질문 세트에서 여러 레이어 동시 처리
- **Graceful Exit**: 매 라운드 후 "계속/정리" 옵션
  - 중간 이탈 → `prd/` 디렉토리에 현재까지 결과 저장
- **"모르겠다"**: 1회→선택지 제시, 3회→리서치/프로토타입 제안, "나중에"→TODO
- **회귀 상한**: 동일 차원 2회, 전체 3회 → 초과 시 TODO 태깅
- 수렴 보드: 매 라운드 시작 시 변경 차원만 업데이트

---

## Phase 3: 기술 인터뷰 (SPEC용)

PRD 인터뷰 완료 후, **기술 상세 인터뷰**를 이어서 진행. 세션 전환 불필요.

**사용자에게 안내**: "제품 기획이 정리되었습니다. 이제 기술 구현 상세를 몇 가지 확인하겠습니다."

기술 인터뷰 영역:

1. **기술 스택** — 이미 정해진 것 vs 선택 필요한 것
2. **아키텍처** — 모놀리스/마이크로서비스, API 스타일(REST/GraphQL)
3. **데이터 모델** — 핵심 엔티티, 관계, 제약조건
4. **인증/보안** — 인증 방식, 권한 모델
5. **인프라/배포** — 호스팅, CI/CD, 환경 분리
6. **트레이드오프** — 시간 vs 품질, 단순함 vs 확장성

**질문 규칙:**
- CPS와 FEATURES.md에서 이미 명확한 것은 질문하지 않음
- 뻔한 것은 스킵 ("TypeScript 쓸까요?" → 이미 결정 가능)
- 한 번에 1-2개 질문
- 복잡도 Low → 3-5개 질문으로 축소
- 복잡도 High → 깊이 있는 인터뷰 (10-15개)

## Phase 4-6: prd-planner 서브에이전트에 위임

인터뷰 전체 완료 후, **prd-planner 서브에이전트를 spawn**하여 문서 생성 위임:

```
Agent 호출:
  subagent_type: prd-planner
  prompt: |
    아래 데이터를 기반으로 prd/ 디렉토리에 문서를 생성해주세요.

    ## 복잡도: [Low/Mid/High]

    ## CPS
    [CPS 내용]

    ## 리서치 결과
    [서브에이전트 A/B/C 결과 요약]

    ## PRD 인터뷰 결과
    [라운드별 질문-답변 요약]

    ## 기술 인터뷰 결과
    [기술 질문-답변 요약]

    ## 수렴 보드 현재 상태
    [수렴 보드 테이블]

    prd/ 디렉토리에 7개 파일을 생성해주세요:
    CPS.md, PRD.md, MARKET.md, USERS.md, FEATURES.md, RISKS.md, SPEC.md, APPENDIX.md
```

prd-planner가 수행하는 작업:
- Phase 4: Six Hats 교차 분석 → PRD.md + MARKET.md
- Phase 5: 전략적 스코핑 → FEATURES.md + RISKS.md + USERS.md
- Phase 6: 기술 명세 → SPEC.md
- 부록 → APPENDIX.md

---

## Phase 7: 자가 검증 (메인 세션)

prd-planner가 생성한 파일들을 읽고 최종 검증:

- CPS(Context-Problem-Solution)와 PRD가 정합하는가?
- 모든 기능에 뒷받침 인사이트가 있는가?
- 위험한 가정 Top 3 명시?
- 안티-기능 정의?
- 실질 지표 (허영 지표 아닌)?
- SPEC에 기술 스택 + API + 데이터 모델이 있는가?
- FEATURES의 모든 기능이 SPEC에서 기술적으로 뒷받침되는가?
- 수렴 보드 전체 🟢 또는 TODO?

검증 실패 시 → prd-planner에 수정 요청 (resume).

완료 후:
```
✅ prd/ 디렉토리 생성 완료 (8개 파일)

📄 CPS.md      — 세계관 정렬
📄 PRD.md      — 코어 전략 + 인사이트
📄 MARKET.md   — 시장 분석
📄 USERS.md    — 사용자 이해
📄 FEATURES.md — 기능 명세
📄 RISKS.md    — 위험 + 성공 지표
📄 SPEC.md     — 기술 명세
📄 APPENDIX.md — 부록

👉 다음 단계: /tth prd/ 읽고 구현해줘
```

## 중요 규칙

- Phase 0.5 CPS 확정 없이 인터뷰 진입 금지
- Phase 1 리서치 + Phase 2 라운드 1은 **반드시 병렬 시작**
- Phase 3 기술 인터뷰는 PRD 인터뷰 직후 **같은 세션에서** 진행
- Phase 4-6은 **prd-planner 서브에이전트에 위임** (메인 컨텍스트 보호)
- 질문은 AskUserQuestion만 사용
- **절대 가정하지 않음** — 모호하면 질문
- 리서치 결과를 질문의 무기로 활용
- 모순 발견 시 정면 지적
- 산출물은 **prd/ 디렉토리**에 분리 저장 (단일 파일 금지)
- 검색: 웹 → Tavily MCP / Gemini CLI, 코드 → Exa MCP, 문서 → Context7 MCP
- Graceful Exit: 언제든 중간 이탈, prd/ 디렉토리에 현재까지 저장
